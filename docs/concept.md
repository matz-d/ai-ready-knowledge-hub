# プロダクトコンセプト

## プロダクト名

**AI-Ready Knowledge Hub**

機能名: **Context Curator**

サブコピー: **SMEの散らばった情報を、AIが使える会社の記憶に変える。**

---

## 提供価値

NotebookLMやGeminiのようなAIツールを、SMEが本当に活用できるようにするための **前段プラットフォーム**。

社内に散らばった文書をAIに渡したいが、

- どの資料を渡してよいか分からない
- 古い資料・新しい資料が混在している
- 個人情報や顧客機密が混入している
- 暗黙知が文書化されていない

という現実的な障壁を解消する。

「整理する」のではなく「変換する」。**AIに渡せない情報を、AIに渡せる形に変える** ことが核心。

---

## ターゲット

**機密文書と暗黙知を多く持つSME**

初期デモ題材:
- 会計事務所・社労士事務所

位置づけ:
- 士業は本作品の唯一の市場ではなく、課題を伝えるための具体的なデモシナリオ
- 本作品の核心は税務・労務などの専門判断ではなく、雑多で機密性の高い社内情報をAIに渡せる形へ変換すること
- 専門業務の正解判定や法的助言は行わず、使える資料・除外すべき資料・不足している暗黙知・人間に確認すべき質問を整理する

デモ題材として士業を選ぶ理由:
- 紙・PDF・Excel・テンプレート・過去相談・法改正資料・顧客別資料・暗黙知が混在しやすく、課題が伝わりやすい
- 機密情報の塊 = マスキング機能の価値が直感的に理解される
- M-Grow AI の事業資産に転用可能 (ハッカソン後も生きる投資)
- 審査員が業界をイメージしやすい (士業は誰もが関わる)

初期デモの想定企業像:
- 従業員12名の会計・社労士合同事務所
- 顧問契約、給与計算、年末調整、就業規則、助成金相談、顧客対応メモ等が PDF / CSV / Excel / テンプレート / 個人メモに散らばっている
- GeminiやNotebookLMを使いたいが、何を入れてよいか分からない
- 古い料金表、過去顧客メモが混在
- 担当者ごとの暗黙知が多い

他業界への横展開例:
- 不動産仲介: 契約書、物件資料、顧客相談メモ、重要事項説明テンプレート
- 製造業: 手順書、品質記録、取引先別仕様、過去トラブル対応メモ
- 医療・介護周辺事業: 業務手順、利用者対応メモ、研修資料、個人情報を含む帳票

## コアフロー

```
1. Dump          : 雑多な資料を一箇所に投入
2. Curator       : AIが自動分類・タグ付け
3. Masker        : 機密度高なら自動マスキング → AI参照可能版を生成
4. Inventory     : 会社の情報資産マップを表示
5. Purpose Query : ユーザが目的を入力
6. Strategist    : AI-ready Context Package を生成
                   (使える情報 / 除外すべき情報 / 足りない情報)
7. Interviewer   : 暗黙知を質問化
```

---

## 4エージェント構成

| エージェント | 役割 | 自律的判断の例 |
|---|---|---|
| **Curator** | 文書分類・タグ付け | 「この文書は古い/新しい?」「機密度は?」「同一文書の別バージョンか?」 |
| **Masker** | PII検出+マスキング | 「Cloud DLP では検出されないが文脈上機密性の高い箇所」を判断、マスク範囲を決定 |
| **Strategist** | 目的→Context Package | 目的→必要情報を計画、Curatorの結果から取捨選択。情報不足を判断したら Interviewer を呼ぶ |
| **Interviewer** | 暗黙知の質問化 | 不足領域から質問を生成、回答を受けて追加質問を判断 |

---

## 機密度4段階

| レベル | 例 | AI参照 |
|---|---|---|
| **Public** | 営業資料、料金表 | そのまま参照可 |
| **Internal** | 業務手順、教育資料 | 社内のみ、参照可 |
| **Confidential** | 顧客名簿、契約書 | **Maskerで変換 → 参照可** |
| **Restricted** | 個人情報の核心、契約秘密の詳細 | マスクしても危険、人間のみ |

`Confidential` のマスキング処理が、本作品の **新価値** の中心。

---

## 物語の核

| | Before (整理プロダクト) | After (本作品) |
|---|---|---|
| 動詞 | 「整理する」 | **「変換する」** |
| 提供物 | 整理されたカタログ | AI参照可能な変換済み資産 |
| 対象 | 元から渡せる情報 | **元は渡せなかった情報も含む** |
| 価値の構造 | カタログサービス | データプロダクト |

「AIに渡せない情報」の扱いが3パターンに分岐する設計:
- 渡せる (Public/Internal) → そのまま使う
- **マスクすれば渡せる (Confidential)** → Masker で変換 ← ここが新価値
- マスクしても渡せない (Restricted) → 人間判断のみ

このうち2番目のカテゴリの存在を見せることで、「AI活用したいけど機密情報が壁」というSMEの代表的悩みに直接的な解を提示する。

---

## ハッカソンでの勝ち筋 (3点)

1. **Purpose Query の知性** : 目的を後から入れると、必要情報・不足情報・質問が自動生成される
2. **AI-ready Context Package** : NotebookLMやGeminiに渡す前の情報セットを作る
3. **Curator eval pipeline** : AIの分類品質をGitHub Actionsで継続評価し、「まわす」を見せる
